# Management console

The console is a local web layer over the existing commands — training,
playing, metrics, and run administration from one page, without changing
how any of those work underneath.

```bash
python -m gomoku_muzero.manager
```

Open the printed address (default `http://127.0.0.1:8000`).

| Option | Default | Meaning |
| --- | ---: | --- |
| `--host ADDR` | `127.0.0.1` | Bind address; local-only by default |
| `--port N` | `8000` | Port to listen on |
| `--checkpoint-dir PATH` | `checkpoints` | Directory scanned for models and states |
| `--device NAME` | `auto` | Compute backend for interactive play |

## Train tab

The options form is generated by introspecting the real training CLI, so
it always matches `python -m gomoku_muzero.train --help` exactly — same
options, same defaults, same help text. The run name is prefilled with a
timestamp default and freely editable. Choosing a training state in the
resume dropdown continues an interrupted run.

Training executes as a subprocess of the real CLI: the console builds
the command line, launches it, and stays a pure observer. The status
card shows the lifecycle state (idle / running / finished / failed), an
iteration progress bar, the latest metrics (loss, policy KL, value
calibration, evaluation scores, throughput), and a live tail of the
training log (kept under `logs/<run-name>.log`).

"Stop" sends the same interrupt as Ctrl-C in a terminal. Because
training saves its checkpoint and training state after every iteration,
stopping loses at most the iteration in progress, and the run reappears
in the resume dropdown.

Only one training run is active at a time.

## Play tab
