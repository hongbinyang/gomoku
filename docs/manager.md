# Management console

The console is a local web layer over the existing commands — training,
playing, metrics, and run administration from one page, without changing
how any of those work underneath.

```bash
python -m gomoku_muzero.manager
```

Open the printed address (default `http://127.0.0.1:8000`).

| Option | Default | Meaning |
| --- | ---: | --- |
| `--host ADDR` | `127.0.0.1` | Bind address; local-only by default |
| `--port N` | `8000` | Port to listen on |
| `--checkpoint-dir PATH` | `checkpoints` | Directory scanned for models and states |
| `--device NAME` | `auto` | Compute backend for interactive play |

## Train tab

The options form is generated by introspecting the real training CLI, so
it always matches `python -m gomoku_muzero.train --help` exactly — same
options, same defaults, same help text. The run name is prefilled with a
timestamp default and freely editable.

Choosing a training state in the resume dropdown continues an
interrupted run — and prefills the whole form from the run that last
wrote that state (found via the run's recorded `config.json`), because
resuming restores only the model, optimizer, replay buffer, and
architecture; runtime options would otherwise silently revert to
defaults. After prefill, set `--iterations` (counted in addition to the
saved counter), optionally a new seed, and leave the run name blank for
a fresh directory. Selecting "fresh run" resets the form to defaults.

Training executes as a subprocess of the real CLI: the console builds
the command line, launches it, and stays a pure observer. The status
card shows the lifecycle state (idle / running / finished / failed), an
iteration progress bar, the latest metrics (loss, policy KL, value
calibration, evaluation scores, throughput), and a live tail of the
training log (kept under `logs/<run-name>.log`).

"Stop" sends the same interrupt as Ctrl-C in a terminal. Because
training saves its checkpoint and training state after every iteration,
stopping loses at most the iteration in progress, and the run reappears
in the resume dropdown.

Only one training run is active at a time.

## Play tab

The unchanged game page from [playing.md](playing.md), served by the
same process at `/play`.

## Metrics tab

Charts render directly in the console from each run's `metrics.jsonl` —
no extra dependencies. Select any combination of runs to overlay; since
resumed runs continue the iteration numbering, selecting a run together
with its resumes draws one continuous trajectory. Charted series: total
loss, policy KL, value loss, value calibration, both evaluation scores,
and games per second (long runs are downsampled for display).

The External tools card integrates the optional `metrics` extra when
installed: start/stop a TensorBoard subprocess over `runs/` with a link
to open it, and generate the static PNG charts for a run
(`losses.png`, `throughput.png`, `evaluation.png`), displayed inline.
Without the extra, both buttons explain what to install.

## Storage tab

Lists every run (iteration range, size, date) and every checkpoint file
(model or resumable training state, size, date).

"Info" on a model shows its basic information: board size and win
length, network architecture (channels, blocks, parameter count),
checkpoint format, total iterations trained, and — cross-referenced from
every run whose configuration wrote that file — the exact training
options used per run. Training states are summarized without loading
their replay buffers.

Deleting requires typing the exact run or file name to confirm, and
anything the active training run is writing (its run directory,
checkpoint, and training state) is refused outright.
